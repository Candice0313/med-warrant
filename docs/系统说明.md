# 医疗证据基础 LLM 评判系统

> **声明：** 本系统所有病例、回答及证据均为合成数据，仅供演示用途，不构成任何医疗建议。

---

## 一、系统是什么

这是一个 **以证据为锚点的 AI 置信度评估系统**。

当一个大语言模型（LLM）对医疗问题给出回答时，它会自称有某种程度的把握（比如 88%）。但这个数字是模型自己说的，没有外部依据。本系统的核心工作就是：**根据实际检索到的文献证据，把模型的置信度往下压到一个合理的天花板，并在必要时强制升级人工审查。**

---

## 二、核心论点

> **证据提升的是天花板，而不是置信度本身。**

具体规则如下：

### 2.1 证据层级与置信度天花板

系统将证据分为六个层级，每个层级对应一个置信度上限：

| 层级 | 说明 | 置信度上限 |
|---|---|---|
| T1 — Systematic Review | 系统评价 / 元分析 | 95% |
| T2 — RCT | 随机对照试验 | 90% |
| T3 — Cohort | 队列研究 / 观察性研究 | 75% |
| T4 — Guideline | 临床指南 / 专家共识 | 80% |
| T5 — Expert Opinion | 专家意见 / 案例报告 | 60% |
| T6 — Model Prior | 模型自身先验（无文献支撑） | 40% |

**规则：** 模型的实际显示置信度 = `min(模型原始置信度, 该层级天花板)`

例：模型说 88% 的把握，但只检索到 T4 指南作为依据 → 最终显示 **80%**（被指南天花板压低）。

### 2.2 个体适用性额外限制

EBM 证据体系评估的是**群体层面**的规律，对于涉及"对这个特定患者是否适用"的声明，系统会额外施加 **60% 的个体范围天花板**，无论文献质量多强。

### 2.3 升级为人工审查的两个条件

| 条件 | 触发 | 严重级别 |
|---|---|---|
| 安全相关声明 + 无文献支撑（T6） | `safety_critical = true` 且 `tier = model_prior` | 🔴 DANGER |
| 显示置信度低于 50% | `displayed_confidence < 0.50` | 🟠 WARNING |

## 四、功能模块

### 4.1 病例评判（Scorecard）

用户选择一个病例后，点击 **Evaluate** 触发评判流程：

1. LLM（或 MockProvider）对每个维度给出置信度 + 推理说明
2. 系统检索文献库，找到最相关的证据
3. Tier Classifier 确定证据所属层级
4. `cap_confidence()` 将置信度压到该层级天花板之内
5. `escalation()` 判断是否需要升级人工审查
6. 结果以卡片形式展示，每张卡片包含：
   - 维度名称 + 安全标签
   - 显示置信度（百分比 + 双色进度条）
   - 升级状态（LOW CONFIDENCE / UNVERIFIED SAFETY）
   - 证据来源芯片（可展开查看原文摘要）

### 4.2 探索模式（Explore Mode）

点击 **⬡ Explore** 开启假设探索模式，**无需任何 API 调用**：

- 每行出现证据层级下拉菜单
- 修改层级后，置信度和升级状态**实时重新计算**
- 行变为红色标注，表示当前显示的是假设值
- 点击 **↩** 可单行恢复原始数据
- 关闭 Explore 模式后，所有行恢复为后端真实数据

**核心演示场景：** 将安全相关维度的证据层级从 T1（系统评价）改为 T6（模型先验），行立即变红并出现 ⚠ UNVERIFIED SAFETY——直观展示"没有证据支撑的安全声明"的危险性。

### 4.3 证据剖面（Profile Tab）

切换到 **◉ Profile** 标签，以雷达图展示所有维度的证据剖面：

- **外圈虚线**：模型原始置信度
- **内圈实心**：被证据压低后的实际置信度
- **顶点颜色**：绿色 = 可信，橙色 = 低置信/升级，红色 = 危险
- 两圈之间的差距 = 证据层级带来的折扣
- 无总分，各维度独立呈现

### 4.4 人工审查队列（Review Queue）

所有触发升级的 verdict 进入队列，审查员可以：

- **Approve**：确认 AI 判断，标记为已审查
- **Override**：添加备注，覆盖 AI 判断

顶部导航栏的红色数字徽章实时显示待审数量。

## 五、关键设计原则

### ❌ 系统绝对不做的事

1. **不展示单一总分** — 把多个维度平均成一个数字，会掩盖其中的安全风险
2. **不让置信度超过证据天花板** — 证据只能降低置信度，不能提高
3. **不让安全声明在没有证据的情况下通过** — 安全 + T6 = 强制人工审查
4. **不隐藏证据来源** — 每个置信度背后的文献都可以展开查看

### ✅ 系统保证的不变量

用 pytest 强制执行，任何破坏这些规则的代码修改都会导致测试失败：

```
1. 显示置信度 ≤ 该层级天花板
2. 显示置信度 ≤ 模型原始置信度（证据不会虚增置信度）
3. safety_critical + model_prior → 必然触发 DANGER 升级
4. individual scope → 额外施加 60% 上限
5. 证据越弱，升级严重程度只增不减
```

## 七、快速启动

```bash
# 后端
cd backend
pip install -r requirements.txt
uvicorn main:app --reload --port 8000

# 前端
cd frontend
npm install
npm run dev
# 访问 http://localhost:5173
```

无需配置任何 API Key —— 默认使用 MockProvider，所有结果均为确定性输出，可重复演示。
